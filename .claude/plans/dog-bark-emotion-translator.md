# 愛犬の鳴き声の感情翻訳ツール — 実装プラン(Phase-wise Gated Plan / 改訂版)

## 前提・確認済み事項

ユーザーへの事前確認により、以下を確定事項として計画する。

| 項目 | 決定内容 |
|---|---|
| 感情カテゴリの粒度 | **6分類 + 棄却クラス(判定不能/犬の鳴き声ではない可能性)**。Phase 1(スパイク)で6分類の見込みを早期検証し、見込みが立たない場合は「3分類(ポジティブ/警戒/不安)+ 棄却」へ縮退する |
| テスト音声の入手方法 | フリー音源サイトのサンプル(Pixabay Sound Effects を優先。帰属表示不要・商用利用可でライセンス管理が容易なため) |
| 精度・成功の判定基準 | 分類の妥当性は主観評価でOK。ただし**特徴量計算の正しさは合成音による自動テストで担保する**(主観評価と自動テストは排他ではない) |
| 対応環境 | PCブラウザ + スマホブラウザ(iOS Safari / Android Chrome)両対応 |

技術方針: Vanilla JS・Web Audio API・Canvas 2D・外部依存なし・全処理クライアントサイド完結(音声データは一切外部送信しない)。README.md / CLAUDE.md は現時点で存在しないため、**Phase 0 で新規作成する**。

## 解析アーキテクチャ方針(重要)

マイク入力とファイル入力の共通化は、次の方式で実現する。

- **特徴量抽出はすべてオフライン解析に統一する**: マイク録音は PCM をバッファ(`AudioWorklet` またはレコーダ経由)に貯め、録音停止後に `AudioBuffer` として解析する。ファイル入力は `decodeAudioData` の結果をそのまま同じ解析関数に渡す。これにより両入力が本当に同一経路になる。
- **`AnalyserNode` + `requestAnimationFrame` は「スペクトログラム表示専用」**とする。rAF ポーリングはフレーム落ちを許容する表示向けの仕組みであり、端末のリフレッシュレート(60Hz/120Hz)で測定値が変わるため、時間精度が必要な特徴量(反復間隔・エンベロープ)の計測には使わない。
- リアルタイム感情判定は行わない。UX は「録音 → 停止 → 解析 → 結果表示」のフローとする(録音中はスペクトログラムのみリアルタイム表示)。

## UI実装方針(重要)

Phase 0〜6 のUIは**機能確認用の仮UI**とし、見た目の作り込みは行わない。機能・情報設計・画面構成がすべて固まった Phase 7 で、外部デザイン案(GPT等で作成した高品質UIデザイン)をHTML/CSSへ落とし込む形で**全面リデザイン**する。

そのため Phase 0〜6 では「後から差し替えやすい構造」を優先する:

- JSからの要素取得は `id` のみ、CSSのスタイリングは `class` のみに分離(CSSを全捨てしてもJSが壊れない状態を維持)
- HTMLは役割ごとに `<section>` で整理し、セクション単位で差し替え可能にする
- UIへの反映処理(状態メッセージ・結果表示等)は関数に切り出し、DOM構造変更の影響を局所化する
- 装飾・アニメーション・アイコン・イラスト等は Phase 7 まで追加しない

## 進め方(重要)

- 本プランは **Phase 0 から Phase 7 まで順番に実装**する。
- **1つの Phase を実装したら、必ずユーザーの承認を得てから次の Phase に進む。** 実装を先読みして複数 Phase をまとめて進めない。
- 各 Phase の「完了条件(検証ゲート)」を満たしたことを Claude が提示し、ユーザーが目視・実機等で確認してから次へ進む。
- Phase の並びは**構築順ではなくリスク順**: 最大の不確実性である「特徴量から感情を分類できるか」を Phase 1 のスパイクで最初に検証し、見込みが立ってから UI・リアルタイム化に投資する。

## Phase 1: スパイク — 分類可能性の最短検証(最大リスクの早期検証)

UI・マイク・スペクトログラムを作り込む前に、**このプロジェクト最大の不確実性「特徴量からルールベースで感情を分けられるか」を最小コストで検証する。**

**作業内容**
- 開発用ページ `test-features.html` を作成(本番UIとは別。以降の Phase で自動テストハーネスに育てる)
- ファイル入力のみで実装: `decodeAudioData` → `AudioBuffer` から直接 FFT・特徴量算出(この時点で `AnalyserNode` は使わない)
- 最低限の特徴量(ピッチ推定・音量エンベロープ・ノイズ成分割合)と、粗い仮分類ロジックを実装
- Pixabay 等から性質の異なる犬の鳴き声サンプルを5〜6種類収集(高く短い吠え、低い唸り、連続吠え、遠吠え風など)し、`test-samples/` に配置
- 全サンプルの特徴量と仮分類結果を一覧表示する

**完了条件(検証ゲート)**
- サンプル間で特徴量に判別可能な差が出ていること(例: 唸り声はノイズ成分が高い、高い吠えはピッチが高い)を一覧表でユーザーと確認
- **Go/No-Go 判断**: 6分類の見込みが立つか、3分類+棄却へ縮退するかをここで決定する(以降の Phase はこの決定に従う)
- 犬の鳴き声以外の音(人の声・音楽など)を1つ入力し、特徴量が犬サンプルと異なる傾向を示すか確認(棄却クラスの実現可能性の当たりを付ける)

**✅ 実施結果(2026-07-08 完了・Go 判断)**
- 合成音テスト全4件 PASS(440Hz正弦波→ピッチ誤差0.001Hz未満、ノイズ/無音/バースト列も期待どおり)
- 実サンプル6件(犬5+ライオン1)で特徴量が明確に分離(ピッチ314〜1520Hz、ノイズ割合0.0007〜0.63、区間数1〜6)。耳での傾向照合もユーザー確認済み
- **判断: Go — 6分類 + 棄却クラスで続行**
- **棄却クラスの守備範囲が実データで3層に確定(Phase 5 の設計と UI 免責に反映すること)**:
  - ①分離可能(見込み): 音楽・長い会話 — 区間構造が犬と根本的に異なるはず。ただし実測は未実施(Phase 5 で長めの会話/音楽サンプルを追加して検証する)
  - ②グレーゾーン: 短い人の発話(hitonokoe.mp3 = 単一区間0.26秒) — ノイズ割合0.68は全サンプル中最高で棄却候補にできるが、ライオン(0.63)・犬の唸りと近接。棄却閾値を攻めると本物の唸り声を誤棄却するトレードオフがある
  - ③分離不能: 近縁動物の咆哮 — ライオンの声(効果音ラボ)は特徴量が犬の唸り領域(ノイズ割合0.63・clarity0.37)と重なり「甘え」に誤分類された。ルールベースでは原理的に区別不能。UI の免責文言に「犬以外の動物の声は正しく判定できません」の旨を含める
- Phase 5 への持ち越し課題: 連続吠え判定の間隔閾値(仮0.5秒)は実測0.65秒に対し厳しすぎた。唸り判定のピッチ条件(250Hz未満)も実測374Hzを取りこぼした。閾値は実サンプルの実測値に合わせて再設計する。棄却判定には clarity(ピッチ信頼度)と区間構造の併用を検討する

## Phase 2: 音声入力基盤(マイク & ファイルアップロード)

**作業内容**
- `js/audio-capture.js` を作成
  - マイク: `getUserMedia` でストリームを取得し、PCM をバッファに貯める(録音停止後に `AudioBuffer` 化してオフライン解析へ渡す)
  - ファイル: `decodeAudioData` でデコードし、同じ解析インターフェースへ渡す
  - **iOS Safari 対策**: ユーザーの操作(ボタンタップ)内で `AudioContext` を生成・`resume()` する(自動再生ポリシー対応)
  - 録音停止時に **`MediaStreamTrack.stop()` で必ずマイクを解放**する。タブ非表示(`visibilitychange`)時も録音を停止・解放する
  - ファイルサイズ上限(20MB)を設け、超過時はデコードせずに案内メッセージを表示する(巨大ファイルによるメモリ枯渇・タブクラッシュの防止)

**完了条件(検証ゲート)**
- PCブラウザ(Chrome)で録音開始→停止し、マイク許可ダイアログ→録音→**停止後にブラウザのマイク使用インジケータが消える**ことを確認
- マイク許可を**拒否**した場合に、エラーで UI が壊れず案内メッセージが出ることを確認(異常系)
- ファイル選択でサンプル(wav/mp3)のデコード成功ログが出ること、**20MB超のファイルが上限チェックではじかれる**ことを確認(異常系)
- **iOS Safari はファイルフォーマットに制約がある(ogg 非対応、mp3/m4a/wav は可)**ことを踏まえ、テストサンプルは mp3/wav で統一されていることを確認
- スマホ実機(Android Chrome)でマイク許可が通ることを確認(`http://<LAN-IP>` アクセス。iOS Safari は HTTPS 制約があるため Phase 6 で mkcert 環境を用意して検証する)

**✅ 実施結果(2026-07-08 完了)**
- ループバック録音テスト(440Hz発振器→AudioWorklet録音チェーン) PASS: ピッチ440.0002Hz・長さ0.502秒。合成音テスト4件も回帰PASS
- 自動検証: 21MB超ファイル拒否 / 破損ファイルの案内表示 / マイク不許可時の案内表示とUI復帰、すべて期待どおり
- PC実機(Chrome): 録音→停止→アドレスバーのマイクバッジ消滅(=MediaStreamTrack解放)・再生をユーザー確認済み。※現行Chromeのマイクインジケータはタブの赤丸ではなくアドレスバー内のバッジ表示
- スマホ実機(Android・iPhone両方): ページ表示・UI確認済み。録音開始時に「安全な接続で開かれていないため…」の案内が正しく表示されることを確認
- **プラン訂正**: Android Chrome も getUserMedia にはセキュアコンテキスト必須で、`http://<LAN-IP>` ではマイク不可(localhost特別扱いはPC上のみ)。スマホの実マイク検証は Android/iPhone とも Phase 6(mkcert HTTPS)に統合する
- **音質改善(ユーザー指摘による)**: getUserMedia のデフォルトDSP(AGC/ノイズ抑制/エコーキャンセル)が音を着色していたため、MIC_CONSTRAINTS で全て無効化。AGCは音量エンベロープ特徴量を、ノイズ抑制は唸り声のノイズ成分特徴量を歪めるため、解析品質にも必須の修正
- **開発環境メモ**: LAN検証URLは `http://192.168.0.62:3000`。Windowsファイアウォールに受信許可ルール `dev-server-3000`(TCP 3000)を追加し、`node.exe` の受信ブロックルール2件(ブロック優先で許可を打ち消していた)を無効化した。**開発終了後に元へ戻す**(ブロックルール再有効化 or dev-server-3000 削除)

## Phase 3: スペクトログラム描画(リアルタイム表示)

**作業内容**
- `js/spectrogram-renderer.js` を作成
- 録音中のリアルタイム表示専用として `AnalyserNode`(`fftSize: 2048` 目安)+ `requestAnimationFrame` で描画(特徴量抽出には使わない — アーキテクチャ方針参照)
- 横軸=時間・縦軸=周波数・色=強度のスペクトログラムを1列ずつ描画・スクロール
- スクロールは `getImageData`/`putImageData` ではなく **`drawImage` による canvas 自己コピー**で実装する(スマホ性能対策)
- **`devicePixelRatio` を考慮した canvas 解像度設計**(ぼやけ防止と描画コストのバランスを取り、必要なら上限 2 でクランプ)
- 録音停止時に rAF ループを確実に解除する

**完了条件(検証ゲート)**
- PCブラウザで高い声/低い声を出し分け、スペクトログラム上で模様の出る高さが変わることをスクリーンショットで比較確認
- ファイル入力のサンプル再生でも鳴き声の輪郭が視覚的に現れることを確認
- 録音停止後に rAF ループが止まっていること(devtools の Performance タブ等で確認)
- スマホ実機(Android Chrome)で描画のカクつき・極端な遅延がないことを目視確認
- **iOS 実機の簡易前倒しチェック**: この時点で一度 iOS Safari でも開き(mkcert 簡易セットアップ or ファイル入力のみで可)、`AudioContext` のサンプルレート等に設計を壊す差異がないか確認する(Phase 6 での手戻り最大化を防ぐ)

**✅ 実施結果(2026-07-08 完了)**
- ピクセル自動検証4件すべてPASS: 低周波ビン(100)→y=115・高周波ビン(800)→y=27(低ビンより上)・スクロールで旧列が1列左へ移動・停止後に描画ループが止まる。回帰(合成音4件・ループバック)も全PASS
- 設計: レンダラーを「周波数データ配列→1列描画」の純粋関数構造と rAF ループ制御に分離したため、音声なしでピクセル単位の検証が可能になった
- PC実機(Chrome): 高い声/低い声の描き分け・停止後の絵の固定・ファイル再生時の輪郭表示をユーザー確認済み
- スマホ実機(iPhone): ファイル再生でスペクトログラム表示・カクつきなしを確認(iOS前倒しチェック完了)
- **iOS の消音スイッチ問題(ユーザー指摘による)**: iOS では Web Audio 出力が既定で ambient 扱いとなり消音スイッチで無音化される。AudioSession API で用途宣言して解決(再生時 `playback` / 録音時 `play-and-record` — 録音中に `playback` を宣言するとマイクが使えなくなるため用途ごとに切替が必要)。非対応環境は feature detection でスキップ
- **スマホのファイル選択問題(ユーザー指摘による)**: `accept="audio/*"` だけだと Android/iOS のピッカーが mp3 をグレーアウトすることがある(MIME登録が octet-stream 等になるため)。拡張子も併記して解決: `accept="audio/*,.mp3,.wav,.m4a,.aac,.ogg"`
- **UI文言の修正(ユーザー指摘による)**: 再生ボタンが常に「録音を再生」でファイル読込時に実態とずれていた。`setPlaybackSource(buffer, kind)` に集約し、初期=「再生」/録音後=「録音を再生」/ファイル後=「音声を再生」へ追従させた

## Phase 4: 特徴量抽出の本実装 + 自動テストハーネス

**作業内容**
- `js/feature-extraction.js` を作成し、Phase 1 のスパイク実装を本実装に整理・拡張:
  - 鳴き声区間の検出(音量閾値ベースの VAD 相当。マイクの AGC でノイズフロアが変動することを考慮し、固定閾値ではなく相対閾値を検討)
  - 基本周波数(ピッチ)の推定値
  - ピッチの時間変化パターン(上昇・下降・平坦)
  - スペクトル重心(音の「明るさ」の指標)
  - 音量エンベロープ(立ち上がりの鋭さ・継続時間)
  - 鳴き声の反復間隔(連続吠えかどうか、間隔の長さ)
  - ノイズ成分の割合(倍音がはっきりしているか、ノイズ的に荒れているか)
  - ピッチ推定(自己相関等)は計算量が大きいため、リアルタイムではなく**鳴き声区間確定後のオフライン処理**として実行する
- `test-features.html` を**自動テストハーネスに昇格**させる:
  - **合成音ユニットテスト**: コードで生成した検証信号に対する自動アサートを実装する — 440Hz 正弦波でピッチ推定が 440±数Hz を返す / ホワイトノイズでノイズ成分割合が高く出る / 無音で鳴き声区間の検出がゼロ件、等。ページを開くと全アサートが実行され green/red で結果表示される
  - サンプル音源の特徴量一覧表示(Phase 1 から継続)

**完了条件(検証ゲート)**
- `test-features.html` の**合成音アサートがすべて green** であること
- サンプル群の特徴量一覧で「高い声はピッチ値が高い」「唸り声はノイズ成分が高い」等、直感と矛盾しない傾向をユーザーと目視確認
- 異常系: 無音ファイル・0秒相当の極短ファイルを入力してもエラーで停止せず「鳴き声区間なし」相当の結果が返ること

**✅ 実施結果(2026-07-08・実マイク確認のみ残)**
- 合成音アサート13件すべてPASS(決定論的・2回実行で同結果)。スペクトログラム4件・ループバック・境界安定性も全PASS。実サンプル7件の回帰チェックも全「一致」
- 追加した特徴量: スペクトル重心(FFT・Hann窓)、ピッチ変化パターン(rising/falling/flat)、noiseFloorRms、peakToNoiseRatio、pitchAgreement
- 解析性能: 5秒の音声を **106ms** で解析(当初703ms/秒 → FFT化で約8.5倍高速化)

**この Phase で発見・修正した重大な設計欠陥(いずれも回帰テストが検出)**
1. **SNRゲートが持続音を検出不能にしていた**: 「peak < ノイズフロア×4 なら鳴き声なし」という判定は、定常的な音(録音全体を埋める遠吠え・正弦波)でノイズフロア≒ピークとなり必ず発動する。RMS分布では「定常ノイズ」と「持続音」を原理的に区別できない(区別できるのは周期性=clarity)。→ ゲートを廃止し、ノイズフロア由来の閾値はピークの50%で頭打ちにした。棄却の判断材料は `peakToNoiseRatio` として分類器へ渡す
2. **ピッチ推定が単一窓で脆かった**: 主区間の中央8192サンプル(0.17秒)だけで代表ピッチを決めていたため、区間境界が0.14秒ずれるとオクターブ誤りで値が倍になった(barking-3: 314Hz⇔681Hz)。→ 複数窓推定 → 1オクターブまでの畳み込み → clarity重み付き中央値、に変更。境界を±0.15秒動かしてもピッチ比1.007に安定
3. **オクターブ畳み込みが多数決を汚染していた**: 基準窓(最大clarity)自体が誤っていると、4オクターブ離れた値まで引き寄せられた(78Hz→1248Hz)。→ 畳み込みは1オクターブまでに制限し、それ以上離れた窓は「不一致」として扱う
4. **唸り声にピッチを与えていた**: 声帯が不規則振動する唸り・咆哮には一意なピッチが存在しない。→ 窓の合意度(`pitchAgreement`)を導入し、合意50%未満なら `pitchHz = null` を返す。**ライオン(barking-6)は合意0.25で null になり、Phase 1 で「区別不能」とされた音に対する新たな棄却手がかりが得られた**
5. 計算量: 自己相関の総当たり O(n×ラグ数) を FFT(ウィーナー=ヒンチンの定理)で O(n log n) に置換

**基準値の更新**: ピッチ推定手法を意図的に変更したため、回帰基準を Phase 4 実測値へ更新(`FEATURE_BASELINE`)。barking-3: 314→319Hz(オクターブ誤りの解消)、barking-6: 373.7→null が主な変化

**ユーザー確認済み(2026-07-08)**: 合成音13件PASS・境界安定性PASS・回帰列すべて「一致」をブラウザ上で確認

**Phase 5 への繰り越し**: 実マイクでの区間検出確認(静かな部屋の環境ノイズを鳴き声と誤検出しないか)。現時点では録音結果を解析して表示する導線がないため、感情分類の結果表示が繋がる Phase 5 のゲートで検証する

**この Phase で判明した設計上の重要事項**
1. **反復構造だけでは「遊び」と「甘え」を分けられない**: bark-2(遊び)は6区間・間隔0.65秒、bark-7/8(甘え)は4〜6区間・間隔0.61〜0.72秒とほぼ同一。分ける軸はピッチ(646Hz vs 1017〜1103Hz)になった
2. **⚠️ ピッチには体格の交絡がある**: bark-7/8 はマルチーズ(小型犬)で、高ピッチは感情ではなく声帯の小ささに由来する可能性が高い。`HIGH_PITCH_BARK_HZ = 900` の閾値のまま大型犬の甘え声を入力すると「遊びの誘い」へ流れるはず。**手持ち9サンプルへの過学習**であり、犬種を跨いだ検証は未実施
3. **⚠️ 棄却ルールの余裕が極めて薄い**: 唸り声(bark-3)と人の声(hitonokoe)を分けているのは noiseRatio 0.579 対 0.636(閾値0.6、余裕わずか0.021)と pitchAgreement 0.58 対 0.50(閾値0.5、境界値そのもの)。**少しノイジーな唸り声は「判定できません」に落ちる**。この2音の音響的な近さは Phase 1 の「グレーゾーン」の指摘どおりで、ルールベースの限界に達している
4. 9/9 の一致は「閾値を合わせた当のデータでの一致」であり、汎化性能の証拠ではない。回帰チェックの用途(閾値変更時に何が壊れたかを検出する)としてのみ有効

**実マイク検証で発見・修正した不具合(ユーザー報告)**
- 症状: 何も喋らず静かな部屋で録音すると「甘え・要求(高い声2000Hz)」と誤判定された
- 真因: 2000Hz は `PITCH_MAX_HZ`(探索上限)そのもので、換気扇等の高域ノイズに対し自己相関が最小ラグに張り付いた結果。意味のある値ではない。さらに Phase 4 で SNRゲートを廃止した際、判断材料(`peakToNoiseRatio`・`peakRms`)を分類器へ渡す設計にしたのに、**分類器側でその判断を実装し忘れていた**
- 修正: 分類器に2つの棄却ガードを追加 —(1)`peakRms < 0.02` なら「音が小さすぎます」(全サンプルの実測は最小0.129で6倍以上の余裕)、(2)ピッチが探索上限の95%以上なら「解析できる範囲を超えた高い音」
- 再発防止: 棄却テストに再現ケースを追加(静かな部屋の環境音=ハム+脈動する微小ノイズ、換気扇相当の高域ノイズ)。両方PASS
- あわせてテストの `Math.random()` をシード付き擬似乱数(mulberry32)に置換。ノイズ耐性テストが実行ごとに合否が揺れる(315.2Hz vs 許容±15Hz)問題を解消し、2回連続実行で同結果になることを確認

**ユーザー確認済み(2026-07-08)**: bark-1〜8 のファイル読み込みで表示される感情と理由が聴感と一致。実マイクで人間の「わんわん」を録音 → 「判定できません(音の高さが安定せず…)」と正しく棄却

**✅ 実機再確認(2026-07-08・Phase 5 完了)**: 静かな部屋で3秒録音 → 「判定できません(解析できる範囲を超えた高い音のため…)」と正しく棄却された。発動したのは音量ガードではなく**ピッチ上限ガード**。AGC無効化により環境音も素の音量で入るため `peakRms` は 0.02 を超えており、周波数側のガードが受け止めた形。2つのガードを別々の根拠で置いたこと(防御の多層化)が効いた

### 専門家資料の調査結果(ユーザー提供・2026-07-08)

サンプル数が9件と少ないため、犬の行動学の一般知見を「事前知識」として閾値設計に取り込む目的で調査した。

**参照した資料**
- [犬の鳴き声でわかる感情と健康サイン(獣医師監修)](https://wanko.peace-winds.org/journal/45515)
- [犬の鳴き声でわかる、愛犬の気持ちや健康状態(ペットライン)](https://www.petline.co.jp/note/dog/habit/001994/)
- [犬の鳴き声でわかる感情サイン](https://ichirin.store/blogs/staff/inuno-nakigoe-kanjo-sign)
- [鳴き声でわかる犬の気持ち](https://www.min-breeder.com/magazine/16947)
- [犬は鳴き声を使い分けている?](https://dog.benesse.ne.jp/withdog/content/?id=216602)

**判明した知見と設計への含意**
1. **音響描写はすべて相対表現**(高い/低い、長い/短い)で絶対Hz値は出てこない。ペットライン記事「気持ちが盛り上がった時に、声のトーンも上がる傾向」が示すとおり、**ピッチは個体内での上下にこそ意味がある**
   → **設計への含意**: 絶対値閾値(`HIGH_PITCH_BARK_HZ = 900`)ではなく、その犬の平常時の声を基準にした**相対ピッチ**で判定すべき。飼い主が愛犬の声を数回録音してキャリブレーションする導線が考えられる(Phase 5 で報告した「体格の交絡」への処方箋)
2. **⚠️ 現行ルールが専門家の知見と矛盾している**: 2記事とも「**高いワンワン = 遊びたい・嬉しい / 低いワンワン = 警戒・拒否**」とする。しかし現行実装は反復吠えを「900Hz以上→甘え・要求 / 未満→遊びの誘い」と分けており逆向き。bark-7/8(小型犬・1017〜1103Hz)を甘えに、bark-2(646Hz)を遊びに合わせた**9サンプルへの過学習**。9/9一致はむしろ過学習の証拠
3. 「甘え・要求」の典型は**鼻を鳴らす「クーン/クンクン」**であり高い吠え声ではない。bark-7/8 は要求吠え(demand barking)で、高ピッチは小型犬という体格に由来する可能性が高い
4. 「アウアウ = 飼い主への呼びかけ・注目要求」という記述が複数記事にあり、**7つ目のラベル「挨拶・呼びかけ」を追加した判断は専門家の分類とも整合**する
5. **資料の限界**: これらは知覚的ラベル(オノマトペ→感情)を扱い、測定量には踏み込まない。唸り声と人の声の余裕0.021といった数値的な問題は資料では解けず、犬種を跨いだラベル付きサンプルが必要

**Phase 7 での活用**: 各感情ラベルに行動学的な補足文(例: 要求吠えは応えると強化されるため、静かになってから構う)を付ける。複数記事で一致する内容に絞る

### UI方向転換: ネオン調 → 暖色イラスト調(2026-07-09・ユーザー判断)

ネオン調(ダークネイビー+シアン)は「デザイナー/プログラマーが見てカッコいいデザイン」に留まり、想定ユーザー(愛犬家、主にF1/F2層)に対する訴求力に欠けるとの指摘。d/フォルダの「女性向け」参考画像(暖色クリーム背景+オレンジ角丸+実写犬)を検討した結果、方向転換を決定。

**意思決定の要点**:
- ネオン版で構築した状態設計・情報設計(id/class分離、パネル構造、ボタンのdisabled/enabled連動、ステータスピル)は「見た目を差し替えても機能が壊れない」ことの実証実験として機能した。今回の転換でも同じ`id`をすべて保持し、機能は無傷
- ヘッダーの実写犬画像は不採用(ユーザー自身の愛犬と見た目が違うと「自分の犬のためのアプリ」という実感が薄れるリスク)。イラスト(犬種を特定しない汎用的な絵柄)で統一する方針に
- 競合6アプリ(ワンリンガル、犬語翻訳等)をApp Store/Play Storeの説明文で調査。「音量と高さだけ」程度の粗い分析手法の記述、無関係機能(GPS散歩記録・年齢計算機等)による水増し、強引なサブスク課金(週¥1,500等)が共通点。本ツールは特徴量抽出+棄却ロジック+判定理由の明示まで作り込んでおり、分析の質そのもので差別化する方針を確認

**実装内容(2026-07-09)**:
1. **配色**: 「女性向け」参考画像からピクセル単位で実測した色を採用(背景`#fdf9f4`、本文`#2d1a12`、アクセント`#fc5c0f`等)。CSS変数を全面更新

3. **「使い方」を4ステップの処理フロー化**: 「①マイクで録音する ②音声を解析する ③感情を推定する ④結果を確認する」の数字バッジ+アイコン+シェブロン区切り。640px以下では縦積みに切替

### 🐛 不具合修正: 咳払いが「遊びの誘い」と誤判定される(2026-07-09・ユーザー報告)

**症状**: 実マイクで咳払いを録音すると「遊びの誘い(テンポよく10回続けて鳴いています)」と表示された。

**原因**: 棄却ルールが「ノイズ比>0.6 **かつ** ピッチ合意度<=0.5」のAND条件だったが、`feature-extraction.js`側で合意度<0.5なら`pitchHz`を既に`null`にしている(`PITCH_MIN_AGREEMENT=0.5`)。そのため`pitchHz !== null`の時点で合意度は必ず0.5以上であり、AND条件の後半はほぼ常に偽になる実質的な死んだコードだった。咳払いは声帯振動を伴うため、ノイズ比が高くても窓間のピッチ合意度は高く出て(荒れているが低音のトーンが一貫している)、棄却をすり抜けて反復回数だけで「遊びの誘い」に分類されていた。

**再現**: 合成音(声帯振動150Hz+乱流雑音、前半トーン優勢→後半ノイズ優勢の包絡、0.25秒×4回)で同じ誤判定(`play_invitation`)を再現できた。ユーザーの実録音そのものは検証できていない(ブラウザ内メモリのAudioBufferのみでディスクに保存されないため、devtools console操作が難しい状況では実データを取得できなかった)。

**修正**: `emotion-classifier.js`の棄却ルールをノイズ比単独の判定に変更(`noiseRatio > REJECT_NOISE_RATIO`のみ)。`REJECT_PITCH_AGREEMENT`定数と未使用になった`pitchAgreement`の分割代入を削除。

**再発防止**: 棄却テストに再現ケース「咳払いの反復」を追加(7件目)。全PASS、既存9サンプルの期待ラベル一致(9/9)・回帰も維持。

**残作業**: ユーザーが再度咳払いを録音できるタイミングで、実際に「判定できません」になることを実機で最終確認する

### 📌 Version 2 の改善項目(現バージョンでは実装しない)

**方針決定(2026-07-08・ユーザー判断)**: 本プロジェクトは研究目的ではなく「ブラウザ上で動作する実用的なWebツールの完成」を優先する。専門家知見との矛盾および将来の改善点は本ファイルへの記録に留め、**当初計画どおり Phase 6・Phase 7 まで完成させる**。完成後、必要であれば以下を独立した改善フェーズ(Phase 5.5)として検討する。

1. **相対ピッチ方式(個体キャリブレーション)の導入**
   - 愛犬の平常時の声を数回録音して基準ピッチを登録し(localStorage 等)、`pitchHz / baselinePitchHz` の比で「高い/低い」を判定する
   - 絶対値閾値(`HIGH_PITCH_BARK_HZ = 900` 等)が体格の交絡を拾う問題への処方箋。専門家資料が音の高さをすべて相対表現で語ることが根拠
   - UI に「この声を愛犬の基準にする」導線が増えるため、UIデザイン確定前に決着させる必要がある

2. **反復吠えの解釈を専門家知見に合わせる**
   - 現行: 反復吠えを「900Hz以上→甘え・要求 / 未満→遊びの誘い」と分岐
   - 専門家知見: 「高いワンワン=遊び・嬉しい / 低いワンワン=警戒・拒否」であり、方向が逆
   - 修正する場合、bark-7/8(小型犬の要求吠え)の期待ラベル再検討が必要。「甘え・要求」の典型は鼻を鳴らす「クーン/クンクン」であり、要求吠えと遊び吠えは音響的に分離できない可能性が高い(両論併記 or クラス統合の検討)

3. **検証方法(採用済み・未実装)**: ピッチシフト不変性テスト
   - 既存サンプルのピッチだけを ±1オクターブ変えても分類結果が変わらないことを自動アサートする
   - 分類が変わるなら、そのルールは感情ではなく体格を見ている証拠になる。新規サンプルを集めずに交絡を検出できる
   - 分類器の単体テストとして「特徴量の pitchHz と baseline を同じ係数で変えても結果が同じ」を検証するのが最も精密(音声のリサンプリングは長さも変えてしまうため)

4. その他: 唸り声と人の声を分ける `noiseRatio` の余裕が 0.021 しかない問題は、犬種を跨いだラベル付きサンプルの収集でしか解けない

## データソース・依存ライブラリ・リスク

### 外部データ(テスト音声)
- **候補**: Pixabay Sound Effects(優先。帰属表示不要・商用利用可で統一されておりライセンス管理が容易)、必要に応じて Freesound.org 等
- **無料/有料**: いずれも無料。ただしライセンスは素材ごとに異なる(CC0 / CC-BY / CC-BY-NC 等)
- **リスクと対応**:
  - ダウンロードした音声ファイルは**リポジトリにコミットしない**(Phase 0 で `.gitignore` に `test-samples/` を追加済み)
  - 帰属表示必須・商用利用不可等の条件があるサンプルは、ローカルでの閾値チューニング用途にのみ使い、成果物やREADMEには含めない
  - iOS Safari の再生互換性のため、サンプルは mp3/wav 形式で収集する(ogg は避ける)

### 依存ライブラリ
- なし。Web Audio API・Canvas 2D API・File API はすべてブラウザ標準機能で、npm 依存・外部API・認証は一切不要(無料・利用制限なし)

### プライバシー・セキュリティ
- 全処理クライアントサイド完結。音声データの外部送信は一切行わず、その旨を UI に明記する(Phase 7)
- 録音停止・タブ非表示時に `MediaStreamTrack.stop()` でマイクを確実に解放する(Phase 2)
- アップロードファイルはサイズ上限(20MB)でガードし、メモリ枯渇を防ぐ(Phase 2)
- HTTPS 配信は mkcert によるLAN内配信のみとし、マイクを扱うページを公開 URL に晒さない(Phase 6)

### 精度・表現に関するリスク
犬の鳴き声から感情を判定する手法は、確立された科学的・獣医学的基準があるわけではない。本ツールの分類ロジックは開発者の仮説とフリー音源サンプルでの照合に基づく**ヒューリスティックな推定**であり、精度を保証するものではない。ユーザーとは「主観評価でOK」という前提で合意済みだが、UI上でも「この判定は参考程度の推定です」といった注記を表示し、誤解を招かない見せ方にする(Phase 5)。また、自信がない入力に対して感情を断定しないよう**棄却クラス**を設ける(Phase 5)。
